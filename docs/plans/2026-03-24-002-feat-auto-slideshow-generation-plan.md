# Auto-Generate Travel Slideshows from Location Data

## Overview

Build a Python CLI that automatically generates TikTok slideshows from the existing place discovery data. Each slideshow showcases the top-rated locations for a city (or category within a city), with AI-generated "iPhone photo" images, text overlays with location names and neighborhoods, and a CTA slide. Supports two hook formats (listicle and story) for A/B testing. Posts as drafts to TikTok via Postiz.

## Problem Frame

Atlasi's discovery pipeline finds trending, non-tourist-trap places from social media data. That data currently lives in the dashboard — it has no marketing output. The goal is to close the loop: use the discovered places to generate viral TikTok/Instagram slideshows that drive app downloads. The Larry 1.0.0 skill (`docs/larry-1.0.0/`) proves the end-to-end approach works (1M+ TikTok views, $670/month MRR) but was built for before/after transformation content. This adapts the proven pipeline for travel location discovery content.

## Requirements Trace

- R1. CLI command generates a complete slideshow from DB data for a given city and optional category
- R2. Two hook formats supported: "listicle" (e.g., "8 places in Tokyo tourists never find") and "story" (person+conflict adapted for travel) — for A/B testing
- R3. Images generated via Gemini Nano Banana Pro API, looking like professional iPhone travel photography
- R4. Text overlays on each location slide showing place name + neighborhood (LLM-enriched)
- R5. CTA slide generated by Nano Banana Pro, replicating an Atlasi app screenshot template with the slideshow's actual places integrated
- R6. Variable slide count (5-15 location slides) to experiment with what performs best
- R7. Manual image override: user can drop in real photos for specific locations instead of AI generation
- R8. Posts to TikTok as SELF_ONLY draft via Postiz API (user adds trending music before publishing)
- R9. Resume/retry: partial runs can be resumed without regenerating completed slides
- R10. Place usage tracking to avoid repetition, with cooldown-based reuse policy
- R11. TikTok 9:16 (1080x1920) format first; Instagram 4:5 as future follow-up

## Scope Boundaries

- NOT building the analytics feedback loop in this plan (that's the next phase — "improve based on analytics")
- NOT implementing Instagram 4:5 format (deferred to follow-up)
- NOT building automated scheduling/cron — manual CLI invocation only
- NOT adding coordinates/geocoding to places (neighborhood is inferred by LLM, not geocoded)
- NOT building a web UI for slideshow management — CLI only
- The discovery pipeline is unchanged; this consumes its output

## Context & Research

### Relevant Code and Patterns

- `discover.py` — CLI entry point pattern: `build_parser()` separate from `main()` for testability, `setup_logging()` with three levels, numbered pipeline steps with `log.info("Step N/M: ...")`, lazy imports, `try/finally` for DB cleanup, implicit resume via state checks
- `pipeline/llm.py` — OpenRouter wrapper with retry/backoff: `LLMError`/`CreditsExhaustedError` hierarchy, exponential backoff (`BASE_DELAY * 2^attempt`), non-retryable errors short-circuit the retry loop, rate-limit 429 handling via `retry-after` header
- `pipeline/extractor.py` — Batch LLM processing: structured JSON output, batch size from config, partial-failure abort (commits completed work, leaves unprocessed items for next run)
- `pipeline/filter.py` — Batch LLM processing with continuation: logs and continues on `LLMError` per batch (does not abort), defensive type coercion for LLM outputs
- `pipeline/db.py` — SQLite helpers; extend with slideshow tracking tables and neighborhood column. Safe migration pattern (ALTER TABLE with duplicate column guard)
- `config.py` — Configuration constants: env vars via dotenv with defaults, single-source-of-truth for derived mappings
- `docs/larry-1.0.0/scripts/add-text-overlay.js` — Reference for proven overlay parameters (6.5% font size, 15% outline width, 75% max width, centered at ~30% from top, safe zones top 10% / bottom 20%). These numeric parameters are the proven element — they transfer identically to any rendering library.

### Institutional Learnings

- Larry's text overlay parameters are battle-tested: 6.5% font size, 15% outline width, centered at 30% from top, 75% max width, safe zones (not top 10%, not bottom 20%)
- Posts MUST go as SELF_ONLY drafts — user adds trending music manually (biggest factor in TikTok reach)
- Postiz release IDs cannot be overwritten — wait 2+ hours before connecting analytics
- The discovery pipeline had category mapping conflicts resolved by single-source-of-truth principle (`docs/solutions/logic-errors/batch-resolve-category-feature-code-review-findings.md`) — apply the same principle to shared data contracts (texts.json schema, meta.json schema)

### External References

- **Gemini Image Gen SDK:** `google-genai` package (NOT deprecated `google-generativeai`). Model: `gemini-3-pro-image-preview` (Nano Banana Pro). Supports 9:16 at 2K resolution (1152x2048). ~$0.134/image standard, $0.067/image batch
- **Gemini Batch API:** 50% cost reduction, JSONL input, up to 24h turnaround. Ideal for generating all slides in one batch
- **Postiz API:** Upload images via `POST /public/v1/upload` (multipart form), create post via `POST /public/v1/posts` (JSON). Rate limit: 30 req/hour. JPG more reliable than PNG for carousels. 1500ms rate-limit buffer between uploads recommended.
- **TikTok travel content:** 7-10 slides is the sweet spot for completion rates. Listicle hooks ("N places in [City] tourists never find") are Tier 1 for travel. Slide numbers (e.g., "3/8") drive completion
- **Text overlay for travel:** Location names ARE the value (unlike Larry's "reactions not labels" approach). Name + neighborhood is the optimal information density
- **CJK font support:** Need a CJK-capable font for Japanese/Korean/Chinese place names (Noto Sans CJK recommended). Pillow loads fonts via `ImageFont.truetype('path/to/font.ttf')` — may need to bundle the font file
- **Pillow text rendering:** Pillow 9.2+ supports `stroke_width` and `stroke_fill` parameters on `ImageDraw.text()`, providing native outlined text equivalent to canvas `strokeText()` + `fillText()`. Text measurement via `draw.textlength()` or `font.getlength()` is functionally equivalent to canvas `measureText()`.

## Key Technical Decisions

- **Pure Python stack (no Node.js subprocess boundary):** Text overlay implemented in Python with Pillow; Postiz posting implemented in Python with `requests`. This decision was reached after evaluating the hybrid alternative (Python orchestrator shelling out to adapted Node.js scripts).

  **Why not the hybrid approach?** The Larry Node.js scripts (`add-text-overlay.js`, `post-to-tiktok.js`) are hardcoded to 6 slides with flat string-array input. Adapting them for this feature requires: variable slide counts, structured JSON input, three layout modes (hook/location/CTA), CJK font support, and slide number rendering. By the time those changes are made, only ~45 lines of the original scripts survive — the `wrapText()` function and the stroke-then-fill draw sequence. Meanwhile, the hybrid approach introduces: a subprocess boundary with coarse error handling (exit codes, not structured errors), a second package manager (`npm` alongside `pip`), a second runtime dependency (Node.js 18+), system-level native dependencies (Cairo, Pango for `node-canvas`), JSON contract drift risk between Python producer and Node.js consumer, and PATH/environment management complexity.

  **Why Pillow works:** Every canvas operation used in Larry's overlay maps 1:1 to Pillow: `loadImage()` → `Image.open()`, `measureText()` → `draw.textlength()`, `strokeText()+fillText()` → `draw.text(stroke_width=N, stroke_fill='black')`, `registerFont()` → `ImageFont.truetype('path')`. Pillow is a self-contained Python wheel — `pip install Pillow` works on all platforms with no system-level dependencies. The proven *parameters* (font sizing, positioning, safe zones) are numeric constants that transfer identically.

  **Why `requests` for posting:** The Postiz API is standard REST (multipart upload + JSON post creation). `requests` is already in `requirements.txt`. The Larry posting script is 116 lines of `fetch` calls — trivially portable. No canvas, WebSocket, or browser-specific APIs are involved.

- **Gemini Nano Banana Pro for image generation:** Best photorealism quality, supports 9:16 at 2K, Python SDK is native. The batch API offers 50% savings for bulk generation. Cost per 10-slide slideshow: ~$1.34 standard or ~$0.67 batch. Model ID: `gemini-3-pro-image-preview` (configured in `config.py` as `GEMINI_MODEL`, following the `OPENROUTER_MODEL` pattern).

- **LLM enrichment for neighborhoods AND image prompts:** A single LLM call per batch of places generates both the neighborhood/district and a detailed visual description for the Gemini prompt. This is cached in the DB to avoid repeated calls. Uses the existing OpenRouter integration.

- **CTA slide via Nano Banana Pro replicating TikTok "Save Place" UI:** The CTA slide replicates the Atlasi app's TikTok integration screen (`docs/Atlasi Ingest 1.PNG` is the reference). The layout is: top half shows the slideshow's hook image embedded as a TikTok video preview (with TikTok logo, @atlasi.app username, and caption snippet), bottom half shows a scrollable place list with the actual location names from the slideshow (names only, no addresses). A yellow "Save N Places" button anchors the bottom. Nano Banana Pro generates this by receiving the reference screenshot as image context + a prompt specifying the exact place names and count. This creates a realistic "save these spots to your trip" CTA that demonstrates the app's value proposition directly.

- **Place usage tracking with 30-day cooldown:** A new `slideshow_places` junction table tracks which places appeared in which slideshows. Places become available again after 30 days. `--allow-reuse` flag overrides.

- **Slide count = location slides:** `--slide-count N` means N location slides. Total slides = 1 (hook) + N (locations) + 1 (CTA) = N+2. Minimum N=4 (ensures total of 6, TikTok's minimum). Maximum N = available non-used, non-tourist-trap places (capped at 15).

- **Gemini error handling follows LLM error hierarchy pattern:** Create `GeminiError` (base) and `GeminiQuotaError` (non-retryable) following the existing `LLMError`/`CreditsExhaustedError` pattern. Content filtering (blocked responses) is treated as non-retryable. Per-slide failures continue to the next slide (following `filter.py`'s continuation pattern), not abort the pipeline.

## Open Questions

### Resolved During Planning

- **Node.js vs Python for overlay?** → Pure Python with Pillow. The Larry scripts' proven *parameters* (font size, positioning, safe zones) are numeric constants portable to any library. The scripts themselves need such heavy rewriting (variable slides, structured JSON, three layout modes, CJK) that reuse benefit is minimal. Pure Python eliminates subprocess boundary complexity, the Node.js runtime dependency, and system-level Cairo/Pango dependencies.
- **Where does neighborhood data come from?** → LLM enrichment step, cached in a new `neighborhood` column on the `places` table.
- **What is the CTA slide?** → Replicates the Atlasi "Save Place" TikTok integration screen (`docs/Atlasi Ingest 1.PNG`). Top half: hook slide as TikTok video preview with @atlasi.app username. Bottom half: place list showing the slideshow's location names (no addresses) with a "Save N Places" button. Generated by Nano Banana Pro with the reference screenshot as image context.
- **Single or dual platform?** → TikTok 9:16 first. Instagram 4:5 as follow-up.
- **What goes on the hook slide?** → AI-generated city establishing shot (skyline, iconic street scene) with hook text overlaid.

### Deferred to Implementation

- **Exact Gemini prompt template for "iPhone photo" quality:** Will need iteration during implementation. Start with the researched template structure ("iPhone 15 Pro, golden hour, shallow depth of field, editorial travel photography") and refine based on output quality.
- **Font file bundling for CJK:** Test Noto Sans CJK availability on the target system; may need to bundle the `.ttf` file in an `assets/fonts/` directory and load via `ImageFont.truetype()`. Fall back to system fonts if bundled font is unavailable.
- **Postiz carousel limit for >10 slides:** The API supports image arrays but needs testing with 12-15 slides to verify no silent truncation.
- **Gemini Batch API vs synchronous:** Start with synchronous for development speed; switch to batch for production cost savings once the pipeline is stable.
- **Pillow stroke_width rendering fidelity vs node-canvas:** Both produce outlined text, but pixel-level rendering may differ slightly. Visually verify during implementation and adjust `stroke_width` ratio if needed (start at 15% of font size per Larry's proven parameter).

## High-Level Technical Design

> *This illustrates the intended approach and is directional guidance for review, not implementation specification. The implementing agent should treat it as context, not code to reproduce.*

```
CLI Entry Point (generate_slideshow.py)
  │
  ├─ 1. Query DB ─────────────────────► places table (top N by virality, filtered)
  │
  ├─ 2. Enrich Places ────────────────► LLM call per batch → neighborhood + image_prompt
  │     (cache in DB)                    (skip if already enriched)
  │
  ├─ 3. Generate Hook ────────────────► LLM generates hook text based on format
  │     (listicle template or
  │      story-arc generation)
  │
  ├─ 4. Generate Images ──────────────► Gemini API (Nano Banana Pro, 9:16, 2K)
  │     ├─ Hook slide: city establishing shot
  │     ├─ Location slides: one per place (or use manual override)
  │     └─ CTA slide: Atlasi app screenshot replica with place names
  │
  ├─ 5. Add Text Overlays ────────────► Python Pillow (pipeline/overlay.py)
  │     ├─ Hook slide: hook text (large, centered)
  │     ├─ Location slides: "Place Name\nNeighborhood" + slide number
  │     └─ CTA slide: minimal or skip if text baked into generation
  │
  ├─ 6. Save Metadata ────────────────► slideshows table + slideshow_places junction
  │
  └─ 7. Post (optional) ──────────────► Python requests (pipeline/posting.py)
        └─ Upload to Postiz → TikTok SELF_ONLY draft
```

Output directory structure:
```
slideshows/
  tokyo-food_and_drink-listicle-2026-03-24-001/
    slide_1_hook_raw.png          # AI-generated, before overlay
    slide_1_hook.png              # After overlay
    slide_2_raw.png               # Location 1, before overlay
    slide_2.png                   # Location 1, after overlay
    ...
    slide_N+1_raw.png             # Last location
    slide_N+1.png
    slide_N+2_cta_raw.png         # CTA, before overlay
    slide_N+2_cta.png             # CTA, after overlay
    meta.json                     # Slideshow metadata: places used, hook, format, config
    post_meta.json                # Postiz posting metadata (created by posting step)
    texts.json                    # Text overlay content for each slide
    prompts.json                  # Gemini prompts used for each slide
```

## System-Wide Impact

- **Interaction graph:** The slideshow pipeline reads from the existing `places` table but does not modify core discovery data. New columns (`neighborhood`, `image_prompt`) are additive — they enrich existing rows without changing the schema contract for the discovery pipeline or dashboard. New tables (`slideshows`, `slideshow_places`) are independent. The `init_db()` function in `pipeline/db.py` gains new migration steps (same safe pattern as category migration). The dashboard (`/api/places`) is unaffected — it does not read `neighborhood` or `image_prompt`, and new tables are not queried.
- **Error propagation:** Gemini API failures use a typed exception hierarchy (`GeminiError`/`GeminiQuotaError`) following the `LLMError` pattern. Per-slide failures continue to the next slide (following `filter.py`). `GeminiQuotaError` stops the entire image generation step (no point continuing without quota). Postiz failures are non-fatal — slides are preserved locally even if posting fails. All errors propagate to the orchestrator which catches them at the step level and reports a summary.
- **State lifecycle risks:** Slideshow creation and place linking are atomic (using `with conn:` transaction pattern from `merge_places()`). Partial image generation is safe due to resume logic (skip existing >10KB files). The `--post` step is the only external side effect and requires explicit opt-in. The `post_meta.json` file is separate from `meta.json` to avoid collision — the orchestrator writes `meta.json` at step 10, the posting module writes `post_meta.json` at step 11.
- **New dependency surface:** `google-genai` adds a network dependency on Google's Gemini API. `Pillow` is a self-contained wheel with no system-level dependencies. Both are well-maintained, widely-used Python packages. No Node.js, npm, or system-level native library dependencies are introduced.

## Risks & Dependencies

- **Gemini image quality for specific locations:** The model may struggle with obscure locations it has limited training data for. Mitigation: The enrichment prompt generates a *type-based* visual description ("a cozy retro kissaten in a quiet Tokyo backstreet") rather than trying to recreate the specific place. The image represents the *vibe*, not a photographic replica. If quality is consistently poor for a location type, the manual override (R7) provides an escape hatch.
- **Gemini content filtering:** Some prompts may be blocked due to safety filters. Mitigation: Content-filtered responses are treated as non-retryable (no point retrying the same prompt). The image gen module logs the blocked prompt and continues to the next slide. If multiple slides are blocked, the orchestrator reports partial results and the user can refine prompts or use manual overrides.
- **CTA slide generation fidelity:** Generating a realistic TikTok "Save Place" UI screenshot from a reference image + place names is experimental. The reference (`docs/Atlasi Ingest 1.PNG`) provides a concrete visual target, but Gemini must: (a) replicate the UI layout (header, video preview, place list, button), (b) embed the hook slide image in the video preview area, (c) render the correct place names as text in the list, and (d) show the correct count in "Save N Places". Text rendering accuracy in AI-generated images is the primary risk — place names may be garbled. Mitigation: iteratively refine the prompt; test with 2-3 variations per slideshow. Manual override (`override_N.png`) is available as fallback. This is the single most experimental part of the plan.
- **Place inventory constraint:** Tokyo has only 16 non-tourist-trap places (13 in food_and_drink). With 10 slides per slideshow and 30-day cooldown, you get 1-2 slideshows before needing to run the discovery pipeline for more cities/categories. This is a content constraint, not a code bug — the pipeline is designed to scale with more data. The `--allow-reuse` flag provides an escape hatch for demo/testing purposes.
- **Postiz rate limiting:** 30 requests/hour. A 12-slide slideshow requires ~13 requests (12 uploads + 1 post creation). Limit of ~2 slideshows per hour. Mitigation: 1500ms buffer between uploads (following Larry's pattern). For bulk generation (multiple cities/categories), generate all slideshows first, then post sequentially with awareness of the rate limit.
- **Pillow stroke rendering vs node-canvas:** Pillow's `stroke_width` parameter may produce slightly different visual output than canvas's `strokeText()` at the same numeric parameters. Mitigation: Visually verify during implementation and adjust the stroke width ratio if needed. The Larry parameters (15% of font size) are a starting point, not a pixel-exact requirement.
- **CJK font availability:** Noto Sans CJK is ~16MB per weight. Mitigation: Bundle the regular weight in `assets/fonts/` and load via `ImageFont.truetype()`. If the font file is missing, fall back to the system default font and log a warning. Place names will render (just with a fallback font) rather than crash.
